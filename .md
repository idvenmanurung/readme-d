# Panduan Demo: `make setup-node-labels-taints`

Dokumen ini dibuat supaya kamu paham **apa yang berubah, kenapa berubah, dan bagaimana cara demoin ke mentor** dari awal sampai selesai — plus antisipasi pertanyaan jebakan.

---

## 1. Konteks Masalah (kenapa task ini ada)

Sebelumnya, kalau client sudah punya cluster K8s yang jalan (node udah join), tapi belum di-label/taint sesuai kebutuhan (misal node khusus buat GPU, database, dll), **tidak ada cara standalone** buat masang label/taint itu.

Script `infra/scripts/setup-node-label-taints.sh` **sudah ada** dan sudah benar bacanya dari `config.yml`, tapi dia **butuh `KUBECONFIG` sudah ter-set duluan**. Kalau belum, script ini gagal karena `kubectl` nggak tau mau connect ke cluster mana.

Solusinya: bikin 1 target Makefile baru yang **otomatis fetch kubeconfig dulu**, baru jalanin label/taint-nya. Jadi user cukup 1x command, tanpa perlu install ulang seluruh infra (`infra-standalone`).

---

## 2. Apa yang Diubah (Code Change)

### a. `Makefile` — target baru

```makefile
# Standalone target: apply node labels/taints on an existing cluster
# (kubeconfig not required beforehand - this target fetches it first)
setup-node-labels-taints:
	@echo "Fetching kubeconfig..."
	bash infra/scripts/setup-kubeconfig.sh
	@echo "Applying node labels and taints from config.yml..."
	bash infra/scripts/setup-node-label-taints.sh
```

**Kenapa strukturnya begini:**
- Baris 1 (`setup-kubeconfig.sh`) → connect ke cluster, hasilnya `KUBECONFIG` di-export supaya `kubectl` di baris berikutnya bisa langsung dipakai.
- Baris 2 (`setup-node-label-taints.sh`) → baca `config.yml`, loop tiap node, apply label & taint.
- Dipisah jadi 2 baris (bukan digabung jadi 1 script baru) karena **reuse script yang sudah ada** — sesuai requirement task (nggak boleh nulis ulang logic yang udah ada).

### b. `Makefile` — entri di `help`

```makefile
@echo "  make setup-node-labels-taints - Fetch kubeconfig and apply node labels/taints from config.yml (standalone, for existing clusters)"
```

Supaya kalau orang lain jalanin `make help`, langsung kelihatan command ini ada dan fungsinya apa.

### c. `GLCHAT_INSTALL_FLOW.md` — **⚠️ ini yang masih perlu kamu tambahin**

Saat aku cek file kamu, tabel **Quick Reference** di `GLCHAT_INSTALL_FLOW.md` **belum** ada baris untuk `make setup-node-labels-taints`. Ini salah satu item di checklist task kamu yang belum selesai. Tambahkan baris baru, contoh:

```markdown

```

Taruh di bawah tabel utama, atau kasih section terpisah "Standalone Utility Commands" biar jelas ini beda dari alur instalasi awal (bukan bagian dari `infra-standalone`).

## 3. Data Node yang Dipakai untuk Demo (dari `config.yml` kamu)

```yaml
nodes:
  - name: worker-1
    labels:
      - workload=app

  - name: worker-2
    labels:
      - workload=db
    taints:
      - workload=db:NoSchedule
```

Jadi target demo-nya:
- `worker-1` → dapat label `workload=app`
- `worker-2` → dapat label `workload=db` **dan** taint `workload=db:NoSchedule`

---

## 4. Step-by-Step Demo (dari nol sampai selesai)

### Step 0 — Pastikan posisi kerja benar
```bash
cd cloud-llm/onprem-kubernetes
```
> Semua command Makefile ini harus dijalankan dari root folder project, karena path script-nya relatif (`infra/scripts/...`).

---

### Step 1 — Cek kondisi node SEBELUM demo (baseline)
```bash
kubectl get nodes --show-labels
kubectl describe node worker-2 | grep -A5 Taints
```
**Tujuan:** biar mentor lihat kondisi "before" jelas — nanti gampang dibandingkan dengan "after".

---

### Step 2 — Hapus label & taint yang ada (simulasi cluster "fresh", belum di-label)

Hapus label:
```bash
kubectl label node worker-1 workload-
kubectl label node worker-2 workload-
```
> Tanda `-` di belakang key label = perintah untuk **menghapus** label itu.

Hapus taint:
```bash
kubectl taint node worker-2 workload=db:NoSchedule-
```
> Tanda `-` di akhir taint spec = **menghapus** taint tersebut.

Verifikasi sudah bersih:
```bash
kubectl get nodes --show-labels
kubectl describe node worker-2 | grep -A5 Taints
```
Pastikan label `workload` dan taint sudah hilang dari kedua node.

---

### Step 3 — (Opsional tapi disarankan) Unset KUBECONFIG lokal

Supaya demo membuktikan bahwa command ini **bekerja walau kubeconfig belum di-set**:
```bash
unset KUBECONFIG
echo $KUBECONFIG   # harus kosong
```

---

### Step 4 — Jalankan command baru

```bash
make setup-node-labels-taints
```

Yang terjadi di balik layar (jelasin ke mentor urut kayak gini):
1. `setup-kubeconfig.sh` jalan → baca `cluster_name` dari `config.yml` → panggil `infra/rancher/get-kubeconfig.sh` → kubeconfig ke-fetch dan otomatis dipakai buat sesi ini.
2. `setup-node-label-taints.sh` jalan → baca `infra.rke2.nodes[*]` dari `config.yml` → loop tiap node → apply label (`kubectl label`) dan taint (`kubectl taint`) satu-satu.

---

### Step 5 — Verifikasi hasil (bukti ke mentor)

```bash
kubectl get nodes --show-labels
kubectl describe node worker-2
```

Expected result:
- `worker-1` → label `workload=app` ✅
- `worker-2` → label `workload=db` ✅ + Taint `workload=db:NoSchedule` ✅
- `master-1` → tidak ada label/taint tambahan (karena di config.yml dia nggak punya field `labels`/`taints`) ✅

---

### Step 6 — Buktikan "graceful handling" untuk node tanpa label/taint

Tunjukkan `master-1` di config nggak punya `labels`/`taints`, tapi script tetap jalan tanpa error — buka log outputnya, harus muncul:
```
Processing node: master-1 (...)
No labels defined for node master-1
No taints defined for node master-1
```
Ini membuktikan acceptance criteria: **"Nodes without labels/taints in config are skipped without error."**

Sekarang aku tambahin 1 target baru di Makefile, `make setup-node-labels-taints`, yang otomatis fetch kubeconfig dulu terus langsung apply label & taint dari config.yml. Jadi cukup 1x command aja, nggak perlu install ulang semua pakai `infra-standalone`. Aku reuse script yang udah ada (`setup-kubeconfig.sh` dan `setup-node-label-taints.sh`), nggak nulis logic baru. ## 6. Pertanyaan Jebakan Mentor + Jawaban & Command

### ❓ Q1: "Kenapa nggak digabung aja jadi 1 script baru, biar simpel?"
**Jawaban:**
> "Karena requirement-nya reuse script yang sudah ada, bg — biar nggak ada duplikasi logic. Kalau nanti ada perubahan cara baca config.yml atau cara apply taint, cukup update di 1 tempat (`setup-node-label-taints.sh`), otomatis kepakai juga di flow `infra-standalone` maupun di command standalone ini."

---

### ❓ Q2: "Kalau `make infra-standalone` dijalanin, apa target baru ini bikin `setup-kubeconfig.sh` jalan 2x atau bentrok?"
**Jawaban:**
> "Nggak bg, karena `setup-node-labels-taints` itu target terpisah, bukan bagian dari urutan `infra-standalone`. Dia cuma dipanggil kalau user explicit ketik `make setup-node-labels-taints`. `infra-standalone` tetap jalan urutan aslinya (deps → load-balancer → rancher → register-nodes → kubeconfig → add-ons → label-taints → gpu-node), nggak ada perubahan di situ."

**Command pembuktian:**
```bash
grep -A10 "^infra-standalone:" Makefile
```

---

### ❓ Q3: "Gimana kalau `setup-kubeconfig.sh` gagal fetch (misal cluster belum ready)? Apakah label/taint tetap coba jalan?"
**Jawaban:**
> "Nggak bg, karena kedua script pakai `set -e` di baris atas. Jadi kalau `setup-kubeconfig.sh` exit dengan error, Makefile langsung berhenti — `setup-node-label-taints.sh` nggak akan dipanggil. Ini penting biar nggak ada kondisi setengah-setengah, misal label ke-apply padahal kubeconfig-nya salah cluster."

**Command pembuktian (`set -e` ada di kedua script):**
```bash
head -n 3 infra/scripts/setup-kubeconfig.sh
head -n 3 infra/scripts/setup-node-label-taints.sh
```

---

### ❓ Q4: "Kalau di config.yml ada node yang namanya salah ketik / node itu belum join cluster, apa yang terjadi?"
**Jawaban:**
> "`kubectl label` / `kubectl taint` bakal error 'node not found', dan karena script pakai `set -e`, proses langsung berhenti di node itu — node-node setelahnya nggak diproses. Ini expected behavior sih bg, karena kalau nama node salah, itu masalah konfigurasi yang harus dibenerin dulu di `config.yml`, bukan sesuatu yang harus di-skip diam-diam."

*(Kalau mentor minta ini di-handle lebih graceful, bisa disebut sebagai potential improvement — bukan bug di scope task ini.)*

---

### ❓ Q5: "Command ini idempotent nggak? Kalau dijalanin 2x, ada efek samping?"
**Jawaban:**
> "Aman bg, idempotent. Baik `kubectl label ... --overwrite` maupun `kubectl taint ... --overwrite` udah pakai flag `--overwrite`, jadi kalau dijalanin berkali-kali hasilnya tetap sama, nggak numpuk atau duplikat."

**Command pembuktian:**
```bash
grep -- "--overwrite" infra/scripts/setup-node-label-taints.sh
```
